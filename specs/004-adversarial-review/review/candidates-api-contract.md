# API / CRD Contract-Stability Candidates — commit cb845dd

Reviewer scope: `api/v1alpha1/*_types.go`, `pkg/adapter/*`, generated `config/crd/bases/*.yaml`
(as generation output), and the ClusterHealth external contract. Controllers were read where
needed to verify that schema claims are honored (Principle IX). All claims below are anchored
to files read in this pass; refutation notes state the strongest counter-argument found.

### API-1: HealthCheck.status.summary MaxLength=1024 can wedge mirroring on long condition messages (high)
- Location: `api/v1alpha1/healthcheck_types.go:83` (marker), `internal/controller/healthcheck_controller.go:209,230-237` (mirror), `internal/controller/addoncheck_controller.go:439` (source)
- Failure scenario: An adapter `Run` fails with a long error. `addoncheck_controller.go:439` writes `readyMessage = runErr.Error()` verbatim into the AddonCheck `Ready` condition — `metav1.Condition.Message` admits up to 32768 chars. `HealthCheckReconciler.mirrorTarget` then copies that message verbatim: `hc.Status.Summary = summarizeFromConditions(target.Status.Conditions)` (line 209), and `summarizeFromConditions` returns the Ready condition's `Message` untruncated. `HealthCheckStatus.Summary` carries `+kubebuilder:validation:MaxLength=1024`, and CRD structural validation applies to status-subresource writes. Any Ready message > 1024 chars makes the API server reject the entire HealthCheck status update; `Reconcile` returns the update error and retries forever with the same payload (deterministic, not transient). The HealthCheck's Result, conditions, and SourceObservedAt all freeze, so the ClusterHealth roll-up keeps serving the *stale previous verdict* for that child indefinitely — exactly the "wrong ClusterHealth contract output" failure class.
- Evidence:
  - `healthcheck_types.go:81-84`: `// +kubebuilder:validation:MaxLength=1024` on `Summary`.
  - `healthcheck_controller.go:230-237`: `summarizeFromConditions` returns `c.Message` with no truncation; grep for `truncat` in `internal/controller/` finds none on this path.
  - `addoncheck_controller.go:427-439`: `readyMessage = runErr.Error()` — unbounded; wrapped client/apiserver errors and probe launch errors routinely exceed 1KiB.
  - Same 1024 bound existed at v0.4.1 (`git show v0.4.1:api/v1alpha1/healthcheck_types.go`), so this is longstanding, not #152 churn.
- Refutation notes: Requires a source condition message > 1024 chars — most controller-authored messages are short, and no test demonstrates an actual >1024 error today. But nothing bounds `runErr.Error()`, error wrapping is additive, and the failure mode is a permanent (non-self-healing) mirror wedge, silent except for reconcile error metrics. `ClusterHealthChildSummary.Summary` (also 1024) is safe only transitively — it copies an already-admitted value. Fix is one truncation helper at the mirror boundary (or at condition-write time).

### API-2: #241 retroactively reserves `warnRatio`/`failRatio` from the adapter-owned thresholds namespace without a ContractVersion bump (high)
- Location: `pkg/adapter/ratio.go:13-29`, `pkg/adapter/version.go:24` (still `"1.0.0"`), `internal/controller/addoncheck_controller.go:858-896`
- Failure scenario: The 1.0.0 contract explicitly deeded the thresholds key space to adapters: `FamilyPolicy.Thresholds` is "intentionally untyped (string-keyed strings) so the contract does not need to evolve when an adapter introduces a new knob" and "Adapter authors document the keys they consume" (`adapter.go:122-141`). An out-of-tree adapter built against contract 1.0.0 (v0.4.1 vintage) that documented its own `warnRatio` knob — with, say, 0.0–1.0 fraction semantics — is now silently double-interpreted: `aggregateWithRatioRollups` (addoncheck_controller.go:858) *replaces that family's per-check contributions in the aggregate fold with the engine's ratio verdict* whenever the key parses, so the persisted `HealthReport.spec.result` and `AddonCheck.status.lastResult` change semantics for pre-existing, previously-valid specs. `EnsureCompatible` cannot catch this: the host still reports contract 1.0.0, so the adapter loads cleanly. `git diff v0.4.1..HEAD -- pkg/adapter/version.go` is empty — the reservation shipped with zero version signal.
- Evidence:
  - `ratio.go:13-19`: "Reserved, engine-level threshold keys (#159). … adapters must not consume or advertise them." — a new normative restriction on a surface the 1.0.0 doc declared adapter-owned.
  - `addoncheck_controller.go:665-667`: reserved keys are exempted from `ThresholdAdvertiser` validation, and `addonCheckPolicy` (line 685-690) still passes them through to the adapter in `Request.Policy` — so the adapter *and* the engine both consume the same key.
  - `version.go:14-24` comment: "additive contract growth (new Request fields, new optional interfaces) no longer rejects adapters" — but key reservation is not additive growth; it changes the meaning of existing inputs.
- Refutation notes: No in-tree adapter ever consumed these keys (`grep -rn warnRatio internal/adapter/` is empty), the operator is pre-1.0 overall, and the key names were chosen to match the percentage shape that #152's CEL already enforces at admission (`addoncheck_types.go:56`), which narrows the realistic collision space. If the project's position is "no out-of-tree adapters exist yet," this degrades to medium. But `pkg/adapter` is the *published* contract with its own SemVer discipline, and the correct spelling of this change under that discipline was a minor bump (1.1.0) plus a release note — the machinery exists (`EnsureCompatible` rejects newer-minor adapters precisely because "the adapter may rely on contract surface this host lacks") and was not used.

### API-3: Top-level `spec` is optional on every CRD, so the CEL immutability contracts are bypassable via remove-then-re-add (medium)
- Location: `api/v1alpha1/healthcheck_types.go:46,109`, `api/v1alpha1/healthreport_types.go:168-169`, `api/v1alpha1/addoncheck_types.go:68,166`; generated: `config/crd/bases/fathom.skaphos.io_healthchecks.yaml` (no top-level `required: [spec]` — verified for all five CRDs)
- Failure scenario: All root types declare `Spec … json:"spec,omitempty"`, so controller-gen emits `spec` as an optional property (only `spec.checkRef`, `spec.addonType`, etc. are required *within* spec). CEL transition rules (`self == oldSelf` on `checkRef`, on `addonType`, on the whole HealthReport `spec`) are ratcheting: they are not evaluated when the field is absent from the new object, nor when it was absent from the old one. Therefore: (1) `kubectl patch healthcheck x --type=json -p '[{"op":"remove","path":"/spec"}]'` admits — the immutability rule never runs; (2) a follow-up update re-adds `spec` with a *different* `checkRef` — old value absent, rule skipped again. Net effect: the documented immutability contracts (SKA-576: "retargeting a wrapper would silently repoint its mirrored status snapshot"; "mutating [a HealthReport] after the fact would rewrite history") are enforceable only against single-step edits. A two-step edit retargets a HealthCheck wrapper or rewrites HealthReport history, defeating exactly the scenario the rules cite.
- Evidence: `grep` over `config/crd/bases/*.yaml` shows `required:` lists only inside `spec`/nested objects; no schema requires `spec` at the object root. The `x-kubernetes-validations` blocks sit on the `spec`/`checkRef` nodes and are transition rules (`oldSelf`).
- Refutation notes: The two-step bypass requires deliberate action by a principal with `update` on the resource, who could often just delete+recreate the object anyway (which is the documented remediation — "replace the HealthCheck instead"). For HealthCheck, delete+recreate is observably different (UID change, status reset), whereas remove/re-add spec *keeps the stale mirrored status attached to the new target* until the next reconcile — that's the corruption window. For HealthReport, RBAC presumably limits writers to the operator. Fix is cheap and compatible: mark `spec` required (`+kubebuilder:validation:Required` / drop `omitempty`) on kinds whose spec carries immutability rules. Standard kubebuilder scaffolding ships this quirk everywhere, so one can argue "matches upstream convention" — but upstream scaffolding doesn't claim spec immutability in its API docs; this repo does.

### API-4: AddonCheck `status.conditions` is an atomic list — missing `listType=map` / patch-merge markers, inconsistent with every sibling kind (medium)
- Location: `api/v1alpha1/addoncheck_types.go:116-119`; generated `config/crd/bases/fathom.skaphos.io_addonchecks.yaml` (conditions array has no `x-kubernetes-list-type`)
- Failure scenario: `HealthCheckStatus.Conditions`, `ClusterHealthStatus.Conditions`, and `NodeCertificateCheckStatus.Conditions` all carry `+listType=map` + `+listMapKey=type` + patch-strategy markers (verified in both Go markers and generated YAML: healthchecks CRD emits `x-kubernetes-list-type: map`, addonchecks CRD emits a bare `type: array`). AddonCheck's conditions therefore merge atomically under server-side apply: any second field manager applying a condition (kstatus-style tooling, a GitOps agent doing SSA on status, a future second controller adding e.g. a `Degraded` condition) replaces or conflicts on the *entire* list rather than merging per-type. This also violates the Kubernetes API conventions for `[]metav1.Condition`, which the rest of this API group follows.
- Evidence: `git show v0.4.1:api/v1alpha1/addoncheck_types.go` shows the same bare markers — longstanding omission, not new churn; the other three kinds gained/have full markers.
- Refutation notes: Today only `AddonCheckReconciler` writes these conditions, via `meta.SetStatusCondition` over the whole list with `Status().Update`, so there is no live multi-owner conflict; the defect is latent. Adding `listType=map` is an allowed, compatible schema change now, and much harder to retrofit after beta. Impact is limited until a second writer exists — hence medium, not high.

### API-5: CheckTargetRef schema docs advertise target kinds the controller rejects — including NodeCertificateCheck, which therefore cannot reach ClusterHealth (medium)
- Location: `api/v1alpha1/healthcheck_types.go:12-14,21` (doc), `internal/controller/healthcheck_controller.go:40,163-176` (enforcement)
- Failure scenario: The CRD field documentation — published via `kubectl explain healthcheck.spec.checkRef` and `docs/reference/api.md` — states CheckTargetRef "references a specialized check resource (AddonCheck, DNSCheck, NodeHealthCheck, NodeCertificateCheck, ReachabilityCheck)". The controller supports exactly one kind: `healthCheckTargetKindAddonCheck = "AddonCheck"`; anything else terminal-fails with `Ready=False/UnsupportedKind` and cleared mirror fields. Three of the listed kinds (DNSCheck, NodeHealthCheck, ReachabilityCheck) do not exist anywhere in the API group (`PROJECT` / `api/v1alpha1` have no such types). The fourth, NodeCertificateCheck, *does exist and produces real results* (`status.lastResult`, HealthReports) — yet a user who follows the schema doc and wraps one in a HealthCheck to get it into a ClusterHealth roll-up gets UnsupportedKind. Consequence: node-certificate health is structurally excluded from the ClusterHealth external contract while the published schema says the opposite (Principle IX honesty violation with real user impact: expired-cert Fail can never surface in the cluster-wide verdict).
- Evidence: no `Enum` on `Kind` (deliberate forward-compat is plausible), but the doc asserts present-tense support; controller comment at line 163 admits "The only supported target kind in v0.1 is AddonCheck."
- Refutation notes: The failure is loud (condition + reason), not silent, and the kind list reads as a roadmap. But schema docs are the user-facing contract — kubectl explain gives no hint that four of five listed kinds fail — and the NodeCertificateCheck→ClusterHealth gap is a genuine capability hole the doc papers over. Fix: correct the doc to "AddonCheck (currently the only supported kind)" or wire NodeCertificateCheck mirroring; either resolves it.

### API-6: `checkRef.apiVersion` is unbounded and silently ignored by the controller (medium)
- Location: `api/v1alpha1/healthcheck_types.go:16-19`, `internal/controller/healthcheck_controller.go:165-183`
- Failure scenario: Two related defects on one field. (1) Validation gap: `CheckTargetRef.APIVersion` has no `MaxLength` (and no format pattern) — the only unbounded string in any spec in the group after the #152 hardening pass (`Kind` 63, `Name` 253, `Namespace` 253; compare `HealthReportTargetRef.APIVersion` MaxLength=253). A megabyte-scale apiVersion admits, inflating etcd objects and the CEL evaluation input of the `self == oldSelf` rule that wraps the struct. (2) Honesty gap: the doc says "APIVersion of the target check resource. When empty, defaults to fathom.skaphos.io/v1alpha1" — implying a non-empty value is honored. `mirrorTarget` never reads `CheckRef.APIVersion`: `checkRef: {apiVersion: example.com/v1, kind: AddonCheck, name: x}` is accepted and silently mirrored from *fathom's* AddonCheck `x`, misattributing a foreign-group ref's status into the ClusterHealth chain with `Ready=True/TargetMirrored`.
- Evidence: grep of `healthcheck_controller.go` shows `CheckRef.Kind`, `.Name`, `.Namespace` consulted; `.APIVersion` never referenced anywhere in `internal/controller/`.
- Refutation notes: In practice nobody has a second `AddonCheck` kind in another group, so the wrong-group mirror is an unlikely deliberate input; and checkRef immutability limits churn. The missing MaxLength is a plain #152 oversight with a compatible one-line fix. Severity medium mostly on the silent-misattribution path; the bound alone would be low.

### API-7: `timeout <= interval` CEL invariant silently unenforced when interval is unset (low)
- Location: `api/v1alpha1/addoncheck_types.go:63` (and the identical rule at `nodecertificatecheck_types.go:25`), `internal/controller/addoncheck_controller.go:50-56,547-559`
- Failure scenario: The rule `!has(self.timeout) || !has(self.interval) || duration(self.timeout) <= duration(self.interval)` short-circuits when `interval` is absent — but `interval` has no schema default; the 5m (AddonCheck) / 1h (NodeCertificateCheck) defaults are runtime-only (`defaultAddonCheckInterval`). So `{timeout: 30m}` with interval unset admits, and the effective configuration violates the documented invariant (30m timeout vs 5m interval): an adapter run may still be executing when its next cadence tick is due. Because controller-runtime serializes reconciles per object the consequence is skewed cadence rather than overlap, but the schema message "timeout must not exceed interval" claims an invariant the API does not actually maintain.
- Evidence: no `+kubebuilder:default` on either `Interval` field; the sibling fields `warnDays`/`criticalDays`/`historyLimit` on NodeCertificateCheck *do* use schema defaults, so the cross-field rule for those (`warnDays >= criticalDays`) is airtight while this one is not — an inconsistent defaulting strategy inside one spec.
- Refutation notes: Moving interval to a schema default would make the rule complete but changes stored-object semantics (persisted `interval: 5m` vs "unset means operator default"), which is itself a contract decision — the runtime-default choice may be deliberate to let the operator change defaults without rewriting objects. Operational impact is minor. Worth an explicit doc line ("the rule is only enforced when both are set") or a `has(interval)`-aware default comparison in CEL.

### API-8: Ratio-threshold field doc misattributes range validation to the adapter; out-of-range value disables the entire check (low)
- Location: `api/v1alpha1/addoncheck_types.go:47-56` (doc + CEL), `internal/controller/addoncheck_controller.go:634-637` (actual enforcement)
- Failure scenario: The Thresholds doc says "The 0-100 range and cross-key semantics stay with the adapter and surface via the Accepted condition." For `warnRatio`/`failRatio` this is wrong twice: (1) the range check is performed by the *engine* (`adapter.ParseRatioThresholds`, reserved keys the adapter must not consume — `ratio.go:13-19`); (2) the consequence of e.g. `warnRatio: "150"` (admitted by the CEL shape pattern `^[0-9]{1,3}([.][0-9]{1,2})?%?$`, which allows up to 999.99%) is `Accepted=False/InvalidPolicy`, which per the code comment "stops the check entirely" — the whole AddonCheck stops running over one out-of-range knob on one family, a substantially bigger blast radius than the doc's "surfaces via the Accepted condition" suggests.
- Evidence: `setAddonCheckAccepted` comment (addoncheck_controller.go:561-566): "an invalid policy outranks the clamp notice because it stops the check entirely."
- Refutation notes: Admitting >100 at admission is a documented CEL-cost tradeoff, sanctioned by the crd-compat allowlist entry for `^.spec.policy.additionalProperties.thresholds` (#152), and the failure is loud on the Accepted condition. This is doc-drift plus a debatable severity policy (whole-check stop vs family skip), not data corruption.

### API-9: AddonCheck and HealthReport have no printcolumns while every other kind does (low)
- Location: `api/v1alpha1/addoncheck_types.go:158-159`, `api/v1alpha1/healthreport_types.go:157-158`
- Failure scenario: `kubectl get addonchecks` shows only NAME/AGE; HealthCheck exposes Result/Kind/Target/Age, ClusterHealth Result/Matched/Age, NodeCertificateCheck Result/Reporting/Desired/Age. AddonCheck has the obvious equivalents (`.status.lastResult`, `.spec.addonType`) sitting unexposed; HealthReport has `.spec.result` / `.spec.observedAt`. Purely operator-UX inconsistency; adding printcolumns is a fully compatible change.
- Evidence: absence of `+kubebuilder:printcolumn` markers on both kinds, present on the other three.
- Refutation notes: Cosmetic by definition; no consumer contract depends on server-side printing. Listed only because the asymmetry looks like an omission rather than a decision.
