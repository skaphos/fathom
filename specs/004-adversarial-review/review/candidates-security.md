### SEC-1: Report-authenticity ValidatingAdmissionPolicy exempts every non-`*-node-agent` writer (medium)
- Location: `internal/controller/nodecertificatecheck_controller.go:438-441` (the `writer-is-node-agent` MatchCondition); consumed by the collect-time check at `internal/controller/nodecertificatecheck_controller.go:748-760`
- Failure scenario: The VAP that binds a per-node report ConfigMap to the writing agent's token node-claim only *applies* when the request user matches `^system:serviceaccount:[^:]+:[^:]+-node-agent$`. Any other principal — a tenant user, or any ServiceAccount with `configmaps` create in the NodeCertificateCheck's namespace whose name does not end in `-node-agent` — is not matched by the policy at all, so the node-name authenticity validation never runs on their write. Such a principal can create a ConfigMap carrying the three report labels (`managed-by=fathom`, `source-kind=NodeCertificateCheck`, `source-name=<check>`) plus a `report.json` and a matching `fathom.skaphos.io/node-name` annotation for an arbitrary node, and the controller will adopt it: the only remaining guard, the collect-time cross-check at line 757, merely requires the annotation to equal `report.Node` — and the forger controls both. Result: a forged Pass hides an expiring control-plane certificate on a real node, or a forged Fail creates a false cluster-health alarm, defeating the anti-forgery goal of #155.
- Evidence:
  - `MatchConditions: [{Name: "writer-is-node-agent", Expression: request.userInfo.username.matches('^system:serviceaccount:[^:]+:[^:]+-node-agent$')}]` — a MatchCondition that evaluates false makes the whole policy skip the request (admits without validation).
  - Collect-time check comment concedes the limit: "On a cluster without the policy an attacker who can write the ConfigMap can set both the payload Node and the annotation to a victim node, so this check alone enforces internal consistency, not authenticity" (`:753-756`). The same is true *with* the policy for any non-node-agent writer.
- Refutation notes: The exemption is deliberate — it lets the operator's own `adoptReportConfigMap` Update (`:782-794`, performed as the operator SA, not a `-node-agent` SA) pass without a node claim. Exploiting it requires `configmaps` create/update in the check's namespace, which is already a meaningful privilege; a tenant who owns the namespace where their own check runs can only mislead their own check's HealthReport. The genuinely adversarial case (#155) — one node-agent forging *another* node's verdict under the shared per-check ServiceAccount — *is* covered, because node-agents do match the condition and are held to their per-pod node claim. So the blast radius is limited to principals who already hold namespace ConfigMap write. Still worth raising because the policy is the security control and this is a whole bypass class the collect-time check cannot close.

### SEC-2: node-agent `/metrics` endpoint is unauthenticated and leaks per-node certificate inventory (medium)
- Location: `cmd/node-agent/main.go:82` and `:120-125` (plain `http.Server` + `promhttp.HandlerFor`, no auth filter); gauge populated with node+path labels at `cmd/node-agent/main.go:163`; gauge defined `internal/metrics/metrics.go:112`
- Failure scenario: The node-agent DaemonSet serves Prometheus metrics over plaintext HTTP on `:8080` with no authentication or authorization — unlike the operator's own metrics server, which gates scrapes behind `filters.WithAuthenticationAndAuthorization` when secure (`internal/app/run.go:142-144`). The `NodeCertificateExpiryDays` gauge is labeled by `node` and by the full on-disk cert `path`. Any pod that can reach the agent pod IP on 8080 can enumerate, for every node, exactly which control-plane/etcd/kubelet certificate files exist and how many days until each expires — a reconnaissance map plus attack-timing signal (e.g. "apiserver cert on node X expires in 3 days"). The only compensating control is the managed NetworkPolicy (`nodecertificatecheck_controller.go:501-541`) restricting ingress to namespaces labeled `metrics=enabled`; that policy is inert on any cluster whose CNI does not enforce NetworkPolicy (flannel default, kindnet, etc.), where the endpoint is reachable cluster-wide.
- Evidence: `srv := &http.Server{Addr: cfg.metricsAddr, Handler: metricsMux(), ...}` with `metricsMux` mounting `/metrics` via `promhttp.HandlerFor(ctrlmetrics.Registry, ...)` and no auth middleware; `metrics.NodeCertificateExpiryDays.WithLabelValues(node, r.Path).Set(...)`. NetworkPolicy comment itself notes enforcement "requires a NetworkPolicy-capable CNI — on clusters without one this object is inert" (`:499-500`).
- Refutation notes: Only expiry *days* and file *paths* are exposed — never key material or cert contents. On a NetworkPolicy-enforcing CNI the surface is closed to all but `metrics=enabled` namespaces. The exposure matches the operator's documented metrics-hardening posture (label-gated), just without the TokenReview/SAR fallback the operator has. Defense-in-depth gap rather than direct compromise.

### SEC-3: node-agent hostPath mounts use `DirectoryOrCreate`, letting a check seed root-owned dirs on every node (low)
- Location: `internal/controller/nodecertificatecheck_controller.go:629-636`
- Failure scenario: The DaemonSet mounts each computed host directory with `HostPathType = HostPathDirectoryOrCreate`. If a configured (allowlist-passing) path's parent directory does not exist on a node, the kubelet *creates* it on the host filesystem as root (mode 0755) at pod start. A principal who can create a NodeCertificateCheck can choose allowlisted-but-absent paths (e.g. `/var/lib/etcd/pki` on nodes that are not etcd nodes, `/etc/etcd/pki` on worker nodes) and cause the operator to seed empty root-owned directories across every selected node — an unnecessary write into privileged host locations on machines that never had those paths.
- Evidence: `hostPathType := corev1.HostPathDirectoryOrCreate` then `VolumeSource: corev1.VolumeSource{HostPath: &corev1.HostPathVolumeSource{Path: dir, Type: &hostPathType}}`. A read-only scanner has no need to create missing directories; `HostPathDirectory` would skip absent paths (which `Scan` already reports as omitted/Skipped).
- Refutation notes: Writes are confined to the operator-approved prefixes (`allowedPathPrefixes`) and to a subset of nodes the check already schedules onto; the created directories are empty and mounted read-only. No file contents are written. Impact is host-filesystem litter / minor integrity, not code execution or disclosure. `DirectoryOrCreate` is also what makes a cross-distribution superset path list not fail pod scheduling, so the choice has a functional rationale.

### SEC-4: User-controlled DNS targets and resolver address drive arbitrary DNS egress from probe pods (low)
- Location: `internal/adapter/coredns/adapter.go:470` and `:388-394` (targets from `dnsTargets(policy)` → probe `-target`); `internal/adapter/nodelocaldns/adapter.go:409` and `:445-456` (`listenAddress` threshold → `DNSNameservers`); probe execution `cmd/probe/main.go:67-99`
- Failure scenario: The `targets` threshold (comma-split, unvalidated) and, for nodelocaldns, the `listenAddress` threshold become the probe binary's `-target` and pinned `-nameserver`. A principal who can create/edit an AddonCheck of type `coredns`/`nodelocaldns` can make an in-cluster probe pod issue DNS lookups for attacker-chosen names (`net.DefaultResolver.LookupHost`), or (nodelocaldns) direct the query at an attacker-chosen resolver IP. This is a DNS-based egress/exfiltration primitive: a lookup of `<encoded-data>.attacker.example` from inside the cluster reaches the attacker's authoritative server, and pinning `listenAddress` to an arbitrary IP turns the probe into a client against a chosen server.
- Evidence: `dnsTargets` returns `strings.Split(policy.Thresholds[thresholdTargets], ",")` with only whitespace trimming — no allowlist or name validation; `runDNSProbe(... DNSNameservers: []string{listenAddress})` where `listenAddress = stringThreshold(policy, thresholdListenAddress, defaultListenAddress)`.
- Refutation notes: Resolving caller-supplied names is the declared purpose of a DNS-resolution check, so arbitrary targets are by-design, not a defect. The probe pod runs hardened (non-root, all caps dropped, `AutomountServiceAccountToken=false`), is created through the *impersonating addon client* so pod creation is bounded by that addon SA's RBAC and namespace, and returns only Pass/Fail plus latency — no response body is exfiltrated back. Realistic impact is limited to DNS-query egress, which most clusters already permit cluster-wide.

Notes on areas examined and cleared:
- Path traversal / hostPath escape: `PathAllowed` (`internal/nodecert/paths.go:73-87`) rejects non-absolute, `..`-containing, and non-prefixed paths, `FilterAllowedPaths` is actually applied in `resolveCertPaths` (`nodecertificatecheck_helpers.go:57-64`) before both the `--paths` arg and `MinimalMountDirs`, and `MinimalMountDirs` refuses `/`. Consistent operator/agent path sets. No escape found.
- node-agent scan robustness: recursion bounded by `maxDirDepth=6` and `maxCertsPerPath=256`; permission errors are Skipped not Error; PEM/x509 parse errors surface as summaries, never file contents. No unbounded walk or content leak.
- Impersonation: SA username is derived from a label-resolved ServiceAccount (`addoncheck_controller.go:476-499`), never from raw spec text — no impersonation-string injection. Clients are direct/uncached so RBAC is actually evaluated (`impersonation/factory.go`), and the in-cluster gate fails closed on empty namespace (`incluster.go`, `run.go:218-221`).
- TLS / serving: HTTP/2 disabled by default (CVE-2023-44487 mitigation, `run.go:105-109`); metrics refuse plaintext-on-network unless explicit `allow_insecure` opt-in (`options.go:195-202`); leader election on by default. No `InsecureSkipVerify` anywhere in scope.
- Probe pod construction: hardened (runAsNonRoot 65532, drop ALL caps, RO rootfs, no SA token, `ActiveDeadlineSeconds`); Sweeper uses a structural `probeShaped` confused-deputy guard before deleting despite cluster-wide pod delete. No privilege issue found.
